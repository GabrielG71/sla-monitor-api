# Conventions

## Package Naming
- Pattern: `com.slamonitor.{service}` (e.g. `com.slamonitor.ingestor`, `com.slamonitor.processor`, `com.slamonitor.alert`)

## Entity IDs
- Always `UUID`, generated by PostgreSQL (`gen_random_uuid()`) — never application-generated UUIDs.

## Timestamps
- PostgreSQL: always `TIMESTAMPTZ`
- Java: always `Instant` (never `LocalDateTime`)

## Error Handling (Kafka)
- `@KafkaListener` uses `DefaultErrorHandler` with exponential backoff
- Unrecoverable errors route to DLT (`raw-checks.DLT`)
- No `@Transactional` on Kafka consumers — offset committed only after successful processing

## Redis Keys
- Always namespaced: `{context}:{identifier}` (e.g. `poll-lock:abc-123`)
- Never use raw IDs as keys

## Flyway
- One concern per migration file
- Never alter existing migrations — only add new versioned files
- Sequential versioning: V1, V2, V3...

## DTOs
- Never expose JPA entities directly from REST controllers
- Use dedicated request/response records (Java `record` type preferred)

## Secrets
- Never commit `.env` files
- Always provide `.env.example` with placeholder values
- Secrets via environment variables only

## Commit Convention
- **Semantic commits, no description body**
- Format: `type(scope): short imperative message`
- Types: `feat`, `fix`, `chore`, `refactor`, `test`, `docs`, `infra`
- Examples:
  - `feat(ingestor): add endpoint registration API`
  - `infra: add docker-compose with kafka and postgres`
  - `chore: scaffold maven project structure`

## Memory Update Rule
- **After every completed task**: commit semantically, then update `memory/` files to reflect the new state (roadmap checkboxes, known-issues, etc.)
